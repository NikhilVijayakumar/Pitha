# MCP-007: Semantic Review Never Executes

**Severity:** Critical
**Affects:** All domain audits (architecture, engineering, feature, feature-technical, philosophy, security, vision)
**Location:** `crates/audit/src/providers.rs` — semantic provider invocation

## Problem

All audit results return empty `semantic_review`:

```json
"semantic_review": {
    "instruction": "",
    "rubrics": {},
    "tasks": []
}
```

The deterministic provider runs fine and produces findings. The semantic provider (LLM-based section and document review) produces nothing.

## Expected Behavior

Per the documentation standard, each domain audit should produce:

1. **Deterministic audit** — structural checks (section presence, heading levels, required subsections)
2. **Semantic audit** — LLM review of each section against rubric criteria:
   - Section-level: content quality, tone, completeness, writing guidance compliance
   - Document-level: coherence, cross-references, technology independence
3. **Summary report** — rolls up deterministic + semantic scores into overall domain score

The semantic audit should produce `tasks` entries for each section/document review, with rubrics, findings, and scores.

## Evidence

### Audit calls with semantic provider

```
audit(domain="architecture", providers=["deterministic", "semantic"])
audit(domain="engineering", providers=["deterministic", "semantic"])
audit(domain="feature", providers=["deterministic", "semantic"])
audit(domain="feature-technical", providers=["deterministic", "semantic"])
audit(domain="philosophy", providers=["deterministic", "semantic"])
audit(domain="security", providers=["deterministic", "semantic"])
audit(domain="vision", providers=["deterministic", "semantic"])
```

### Results

All return:
- `score.overall`: 100.0 (misleading — only section presence)
- `score.bucket_scores`: deterministic scores only (e.g., `architecture_deterministic_section: 87.89`)
- `semantic_review.tasks`: empty array
- `semantic_review.rubrics`: empty object
- `semantic_review.instruction`: empty string

### What should exist

For architecture domain with 9 documents, the semantic review should produce:

```
tasks: [
  {
    "document_id": 1,
    "section_id": 45,
    "section_heading": "Component Model",
    "semantic_type": "component_model",
    "rubric": { "completeness": "...", "tone": "...", "technology_independence": "..." },
    "findings": [...],
    "score": 85.0
  },
  ...
]
```

## Impact

1. **Incomplete audit scores** — Current scores only reflect structural presence, not content quality
2. **No rubric-based evaluation** — The detailed writing guidance in standards (tone, voice, structure, audience) is never checked
3. **No section-level findings** — Can't identify which specific sections need improvement
4. **Summary report incomplete** — `get_summary_report` returns `spec_score: null` and `standard_score: null` because semantic data doesn't exist

## Root Cause Hypothesis

The semantic provider is either:
1. Not being invoked despite being passed in `providers` array
2. Invoked but failing silently (no error returned)
3. Invoked but not producing tasks (empty response from LLM)
4. Configuration missing — no rubric/evaluation criteria passed to the provider

## Fix Required

1. Verify the semantic provider is actually invoked when `providers=["deterministic", "semantic"]`
2. Add error logging when semantic provider fails or returns empty
3. Implement section-level rubric evaluation using the standards' writing guidance
4. Populate `semantic_review.tasks` with per-section findings
5. Update `get_summary_report` to incorporate semantic scores
