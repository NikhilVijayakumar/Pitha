# STD-011: Engineering Collection-Level Checks Fire on Individual Documents

**Domain:** engineering
**Severity:** High
**Impact:** Artifically deflates audit score from ~95+ to 72.20
**Status:** Open

## Description

Engineering audit checks (e.g., `eng-sec-guiding-001`, `eng-sec-build-001`, `eng-sec-rationale-001`) are defined at document scope but evaluate collection-level requirements. Every check fires against all 6 engineering documents instead of evaluating the collection as a whole, producing 106 findings for documents that all have the required sections.

## Affected Checks

All `eng-sec-*` section checks fire 6× (once per document) instead of 1× (once per collection):

- `eng-sec-guiding-001/002` — Guiding Principles
- `eng-sec-build-001/002/005` — Build Standards
- `eng-sec-rationale-001` — Rationale
- `eng-sec-code-001/002` — Code Standards
- `eng-sec-constraints-003` — Constraints
- `eng-sec-hook-001/002/003` — Hook Conventions
- `eng-sec-pat-001/002/005` — Component Patterns
- `eng-sec-state-001/002/004/005` — State Management
- `eng-sec-trace-002/003` — Traceability
- `compile-missing-section` — Reports missing sections that exist

## Root Cause

Check definitions in `audit/deterministic/section/07-engineering/*.yaml` use document scope when they should use collection scope. The standard itself states: "Engineering quality is evaluated across the complete Engineering Documentation collection."

## Evidence

All 6 engineering documents have the 4 required sections (guiding_principles, rationale, build_standards, testing_standards) plus optional sections (purpose, code_standards, constraints, traceability). Content quality is high with detailed, project-specific standards. The 72.20 score is entirely from inflated findings, not missing content.

## Recommended Fix

Update check definitions to use collection scope, or add a `scope: collection` flag to section-level checks that represent collection-wide requirements.
